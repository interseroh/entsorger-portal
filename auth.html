1. Anfordern eines OAuth Token

  • Location:
    • Direct:  https://host:8443/oauth/token  
    • Proxy:  https://host/oauth/token
  • Für weitere Informationen zu OAuth 2.0 siehe  https://tools.ietf.org/html/rfc6749
  • Das OAuth 2 Access-Token wird beim Aufruf des Bestelldaten-Services benötigt.
    • Für Authorization Basic: user --> client_id und password --> client_secret. Achtung:
      • "YmVjbzpiZWNvMjAxNQ==" ist dabei der Base64-kodierte String 'beco:beco2015‘, also die client_id ' beco ’und das client_secret ‘ beco2015 ’ durch ein ‘:’ getrennt.
    • Für die Authentifizierung am OAuth-Server verwenden wir 'Password-Credentials' als Grant-Type (Benutzername und verschlüsseltes Passwort von Oracle E-Business Suite bzw. iSupplier). Beispiel Authentifizierungsdaten:
      • User: Benutzername aus dem iSupplier
      • Verschlüsseltes Password: PASS (ACHTUNG: Verschlüsseltes Passwort!)

Anfrage Access Token

https://host:8443/oauth/token
   
POST / HTTP/1.1
Host: server.example.com
Authorization: Basic YmVjbzpiZWNvMjAxNQ==
Content-Type: application/x-www-form-urlencoded
grant_type=password&username=user&password=pass

CURL-Befehl

curl -H "Authorization: Basic YmVjbzpiZWNvMjAxNQ==" --form "grant_type=password" --form "username=user" --form "password=pass" https://host/oauth/token

Antwort mit Access Token

HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache
{
     "access_token":"feefdae9-61ef-434a-8e0c-85eab2dcb02a",
     "token_type":"bearer",
     "expires_in":1800,
     "refresh_token":"a7d6d617-cc39-4379-bceb-55cc4a3a8412"
}

Falls die Zugangsdaten des Benutzers fehlerhaft sind, wird eine Fehler-Nachricht zurück geliefert.

Error Anfrage Access Token

{
     "status":401,
     "error":"Unauthorized",
     "message":"Error description ...",
     "path":"/oauth/token"
}


2. Ansprechen der API-Schnittstelle der Bestelldaten

  • Location: 
    • Direkt: http://host:9017/ep-bestellung/v1/bestellungen
    • Proxy: http://host/ep-bestellung/v1/bestellungen

Anfrage

Übergabe der Daten erfolgt durch POST eines JSON-Dokuments an die REST-Schnittstelle.

Der Aufbau des JSON-Dokuments wird wie folgt vorgegeben. Hierbei ist die Angabe der "itemId" (Datenbankfeld: CATEGORY_ID) optional:

http://host/ep-bestellung/v1/bestellungen
 
POST / HTTP/1.1
Host: server.example.com
Content-Type: application/json
Authorization: Bearer feefdae9-61ef-434a-8e0c-85eab2dcb02a
{
     "requestDate":"20160218",
     "itemId":"29193",
     "vendorId":"7664",
     "requestDateEnd":"20160220",
     "standort":"6612"
}    

CURL-Befehl

  • Hierbei ist die Angabe der "itemId" (Datenbankfeld: CATEGORY_ID) , der "vendorId" und des Kennzeichens "standort" optional. Mindestens ein Datum ("requestDate" oder "requestDateEnd") muss gefüllt sein:
In Windows curl:
curl -H "Content-Type: application/json" -H "Authorization: Bearer 1d9bbf4d-8462-4440-ad77-6447e0b71521" -X POST -d "{\"requestDate\":\"20160218\",\"itemId\":\"29193\"}" https://host/ep-bestellung/v1/bestellungen
 
In Linux curl:
curl -H "Content-Type: application/json" -H "Authorization: Bearer 1d9bbf4d-8462-4440-ad77-6447e0b71521" -X POST -d '{"requestDate":"20160218", "vendorId":"7664", "requestDateEnd" :"20160220", "standort" : "6612", "itemId" : "29193" }' https://host/ep-bestellung/v1/bestellungen
