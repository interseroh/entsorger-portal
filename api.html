1. Generelles

In diesem Abschnitt stehen Bedingungen, die alle Anfragen an die API betreffen.


1.1 Authentifizierung

Alle RESTful-Abfragen dieser API setzen voraus, dass ein gültiges OAuth-Token übergeben wird. Wie ein solches Token generiert wird ist hier beschrieben. Dieser Token wird dann, wie hier beschrieben, in den Header des REST-Requests eingefügt.

Einen Test-OAuth-Token kann man unter der URL https://interseroh-portal-test.com/oauth/token erhalten.
In der Produktivumgebung lautet die URL https://interseroh-host/oauth/token.


2. Bestellungen

In diesem Abschnitt werden API-Aufrufe beschrieben, die den Abruf von Bestellungen zum Gegenstand haben.

Wenn mehrere Bestellungen in der Response zurückgegeben werden, sind die Bestellungen zeitlich sortiert. D.h. die jeweis neueste Bestellung steht an erster Stelle.

Mit dem Test-OAuth-Token kann testweise auf die https://interseroh-portal-test.com/entsorger-bestellung/v2/bestellungen zugreifen.

Eine Online-Dokumentation zum genauene Aufbau des Request und der Response ist unter der URL http://host.interseroh-ag:9101/entsorger-bestellungen/v2/api-docs?group=bestellung verfügbar.


2.1 Alle Bestellungen abrufen

Request

Das Abrufen von Bestellungen wird als GET-Request durchgeführt. der Aufruf gestaltet dich so:

https://www.interseroh-portal.de/entsorger-bestellung/v2/bestellungen?maxanzahl=xxx&mindatum=dd.mm.yyyy

Wird der optionale Parameter maxanzahl angegeben, ist xxx die Anzahl der Bestellungen, die man maximal abrufen möchte. Wird der Parameter maxanzahl nicht angegeben, werden 100 Bestellungen zurückgegeben.

Wird der optionale Parameter mindatum angegeben, so werden nur Bestellungen ab dem angegebenen Datum berücksichtigt. Wird der Parameter mindatum nicht angegeben, werden alle Bestellungen berücksichtigt.


Response

Bei einem Fehler wird der entsprechende HTTP-Statuscode zurückgesendet. Ist der Statuscode 200, befinden sich die Nutzdaten im Body der Response im JSON-Format.

Im Fall eines Erfolges wird ein JSON zurückgegeben, der eine Liste der gefundenen Bestellungen entfällt. Jede Bestellung hat den folgenden Aufbau:

{
   "ticketId": "string",
   "bestellnummer": "string",
   "faelligkeitsdatum": "yyyy-MM-dd'T'hh:mm:ssZ",
   "erstellungsdatum": "yyyy-MM-dd'T'hh:mm:ssZ",
   "status": "enum",
   "mitteilung": "string",
   "lieferant": {
     "nummer": "string",
     "betriebsstaette": {
       "kennung": "string"
     }
   },
   "rahmenvertrag": {
     "name": "string"
   },
   "anfallstelle": {
     "nummer": "string",
     "postalischeAnschrift": {
       "adressatenname": "string",
       "gebaeudeadresse": {
         "strassenname": "string",
         "postleitzahl": "string",
         "ortsbezeichnung": "string"
       }
     }
   },
   "hauptposition": {
     "nummer": "string",
     "bestellteMenge": 0,
     "mengeneinheit": "string",
     "artikel": {
       "nummer": "string",
       "beschreibung": "string"
     },
     "avvnummer": "string"
   },
   "optionsbuendel": [
     {
       "nummer": "string",
       "bestellteMenge": 0,
       "mengeneinheit": "string",
       "artikel": {
         "nummer": "string",
         "beschreibung": "string"
       },
     }
   ]
}

Die ticketId ist ein eindeutiger Kennzeichner für die Bestellung. Diese TicketId wird auch bei einer Leistungsmeldung verwendet, um diese Leistung der zugrundeliegenden Bestellung zuordnen zu können.

Die bestellteMenge ist immer eine Fließkommazahl. Anstelle eines Kommas wird hierbei ein Punkt verwendet.

Der status ist eine Enummeration. er kann die Werte "Offen", "Disponiert", "Rueckgemeldet" und "Fakturiert" annehmen.



3 Leistungen

In diesem Abschnitt werden API-Aufrufe beschrieben, die die Rückmeldung einer Leistung zum Gegenstand haben.

3.1 Leistungsmeldung Typ 1

Bei der Leistungsmeldung Typ 1 handelt es sich um eine Leistungmeldung auf eine vorher abgerufene Bestellung. Die TicketId der Bestellung ist bekannt.

Request

Die Leistungsrückmeldung wird als POST-Request übertragen, wobei sich die Leistungsmetadaten als JSON-String im Body des Requests befinden. Sie haben das folgende Format:

{
   "ticketId": "string",
   "verwogen": true / false,
   "anfallstelle": {
     "nummer": "string",
   },
   "leistungsnachweis": {
     "lieferscheinnummer": "string",
     "datum": "yyyy-MM-dd'T'hh:mm:ssZ",
     "menge": 0.0,
     "mengeneinheit": "string",
     "artikel": {
       "nummer": "string"
     },
     "behaelter": {
       "menge": 0,
       "typ": "string"
     }
   }
}

Die TicketId entspricht der TicketId, der zugehörigen Bestellung.

Verwogen muss auf true gesetzt sein, wenn ein Wiegebeleg mitgeliefert wurde. Ist es auf false gesetzt oder gar nicht angegeben, wurde kein Beleg mitgeliefert.

Das Datum des Leistungsnachweises enthält datum und Uhrzeit der tatsächlichen Leistungserbringung.

Die Menge des Leistungsnachweises ist immer eine Fließkommazahl. Anstelle eines Kommas wird hierbei ein Punkt verwendet.

Die Menge der Behälter ist immer eine Ganzzahl.


Response

Die Response besteht aus einem der Statuscodes:

  • 201 - Created
  • 400 - Bad Request
  • 401 - Unauthorized
  • 403 - Forbidden
  • 404 - Not Found
  • 500 - Internal Server Error
Die genauere Beschreibung eines Fehler befindet sich im Response-Body.
