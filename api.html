1. Generelles

In diesem Abschnitt sind Punkte aufgeführt, die alle Anfragen an die API betreffen.


1.1 Fachliche Beschreibung

Aus dem Auftrag eines Kunden, der bei Interseroh eingeht, wird eine Bestellung an einen Entsorger generiert. Der Entsorger ruft die Bestellung ab, erbringt seine Leistung entsprechend der Bestellung und meldet diese Leistung an die Interseroh zurück. Er kann den Status seiner Leistungsmeldung anschließend abrufen.


1.2 Authentifizierung

Alle RESTful-Abfragen dieser API setzen voraus, dass ein gültiges OAuth-Token übergeben wird. Wie ein solches Token generiert wird, ist hier beschrieben. Dieses Token wird dann, wie an dieseer Stelle beschrieben, in den Header des REST-Requests eingefügt.

Einen Test-OAuth-Token kann man unter der URL https://interseroh-portal-test.com/oauth/token erhalten.
In der Produktivumgebung lautet die URL https://[interseroh-host]/oauth/token.

1.3 Prozessmodell

Das Prozessmodell bildet den Prozess von der Abfrage einer Bestellung bis zur der Leistungsrückmeldung über die API in BPMN 2.0 ab.



2. Bestellung

In diesem Abschnitt werden API-Aufrufe beschrieben, die den Abruf von Bestellungen zum Gegenstand haben.

Wenn mehrere Bestellungen in der Response zurückgegeben werden, sind die Bestellungen zeitlich sortiert. D.h. die jeweils neueste Bestellung steht an erster Stelle.

Mit dem Test-OAuth-Token kann testweise auf die https://interseroh-portal-test.com/entsorger-bestellung/v2/bestellungen zugreifen.

Eine Online-Dokumentation zum genauene Aufbau des Request und der Response ist unter der URL http://[interseroh-host]/entsorger-bestellungen/v2/api-docs?group=bestellung verfügbar.

2.2 Alle Bestellungen abrufen

Request

Das Abrufen von Bestellungen wird als GET-Request ausgeführt. Der Aufruf der URL gestaltet sich derart: https://[interseroh-host]/entsorger-bestellung/v2/bestellungen?maxanzahl=xxx&mindatum=dd.mm.yyyy

Wird der optionale Parameter maxanzahl angegeben, ist xxx die Anzahl der Bestellungen, die man maximal abrufen möchte. Wird der Parameter maxanzahl nicht angegeben, werden maximal 100 Bestellungen zurückgegeben.

Wird der optionale Parameter mindatum angegeben, so werden nur Bestellungen ab dem angegebenen Datum berücksichtigt. Wird der Parameter mindatum nicht angegeben, werden alle Bestellungen berücksichtigt.

Response

Bei einem Fehler wird der entsprechende HTTP-Statuscode zurückgesendet. Ist der Statuscode 200, befinden sich die Nutzdaten im Body der Response im JSON-Format. Im Fall eines Erfolges wird eine Liste der gefundenen Bestellungen im JSON-Format zurückgegeben. Jede Bestellung hat den folgenden Aufbau:

{
   "ticketId": "string",
   "bestellnummer": "string",
   "faelligkeitsdatum": "yyyy-MM-dd'T'hh:mm:ssZ",
   "erstellungsdatum": "yyyy-MM-dd'T'hh:mm:ssZ",
   "status": "enum",
   "mitteilung": "string",
   "auftrag": {
     "turnus": "boolean",
     "uniqueKey": "string"
   },
   "betriebsstaette": {
     "kennung": "string",
     "uniqueKey": "string",
     "lieferant": {
       "nummer": "string"
     }
   },
   "rahmenvertrag": {
     "name": "string"
   },
   "leistungsadresse": {
     "nummer": "string",
     "postalischeAnschrift": {
       "adressatenname": "string",
       "gebaeudeadresse": {
         "strassenname": "string",
         "postleitzahl": "string",
         "ortsbezeichnung": "string"
       }
     }
     "typ": "enum",
     "uniqueKey": "string"
   },
   "positionen": [
     {
       "nummer": "string",
       "menge": 0,
       "masseinheit": "string",
       "artikel": {
         "nummer": "string",
         "beschreibung": "string",
         "uniqueKey": "string",
         "avvnummer": "string"
       },
       "typ": "enum"
     }
   ]
}

Die ticketId ist ein eindeutiger Kennzeichner für eine Bestellung. Diese TicketId wird auch bei einer Leistungsmeldung verwendet, um sie der zugrunde liegenden Bestellung zuordnen zu können.

Die menge ist immer eine Fließkommazahl. Anstelle eines Kommas wird hierbei ein Punkt verwendet.

Der Status ist ein Aufzählungstyp. Er kann die Werte "Offen", "Disponiert", "Rueckgemeldet", "Fakturiert" sowie "Storniert" annehmen.


2.3 Einzelne Bestellung abrufen

Request

Das Abrufen einer einzelnen Bestellung wird ebenfalls als GET-Request realisiert. Dieser Aufruf sieht folgendermaßen aus: https://[interseroh-host]/entsorger-bestellung/v2/bestellungen/ticketId

Bei diesem Aufruf ist es notwendig, die beim Abruf aller Bestellungen mitgelieferte ticketId anzugeben, da diese systemübergreifend eindeutig ist.

Response

Bei einem Fehler wird der entsprechende HTTP-Statuscode zurückgesendet. Ist der Statuscode 200, befinden sich die Nutzdaten im Body der Response im JSON-Format.

Im Fall eines Erfolges wird die gefundene Bestellung im JSON-Format zurückgegeben. Der Aufbau ist analog dem Abruf aller Bestellungen.

3.1 Domain Modell

3.2 Leistungsmeldung Typ 1

Bei der Leistungsmeldung Typ 1 handelt es sich um eine Leistungmeldung auf eine vorher abgerufene Bestellung. Die ticketId der Bestellung ist bekannt.

Request

Die Leistungsrückmeldung wird als POST-Request übertragen, wobei sich die Leistungsmetadaten als JSON-String im Body des Requests befinden. Sie haben das folgende Format:

{
   "ticketId": "string",
   "verwogen": true / false,
   "anfallstelle": {
     "nummer": "string",
   },
   "leistungsnachweis": {
     "lieferscheinnummer": "string",
     "datum": "yyyy-MM-dd'T'hh:mm:ssZ",
     "menge": 0.0,
     "mengeneinheit": "string",
     "artikel": {
       "nummer": "string"
     },
     "behaelter": {
       "menge": 0,
       "typ": "string"
     }
   }
}

Die TicketId entspricht der ticketId der zugehörigen Bestellung.

Verwogen muss auf true gesetzt sein, wenn ein Wiegebeleg mitgeliefert wurde. Ist es auf false gesetzt oder gar nicht angegeben, wurde kein Beleg mitgeliefert.

Das Datum des Leistungsnachweises enthält Datum und Uhrzeit der tatsächlichen Leistungserbringung.

Die Menge des Leistungsnachweises ist immer eine Fließkommazahl. Anstelle eines Kommas wird ein Punkt verwendet.

Die Menge der Behälter ist immer eine Ganzzahl.

Response

Die Response besteht aus einem der Statuscodes:

  • 200 - Ok
  • 400 - Bad Request
  • 401 - Unauthorized
  • 403 - Forbidden
  • 404 - Not Found
  • 500 - Internal Server Error
Die genauere Beschreibung eines Fehler befindet sich im Response-Body.




4. Stammdaten

In diesem Abschnitt werden API-Aufrufe beschrieben, die den Abruf von Stammdaten zum Gegenstand haben.

Es gibt vier Arten von Stammdaten: LIEFERANT, ARTIKEL, ANFALLSTELLE und KOMBINATION. Die Art wird im GET-Request mit dem Parameter typ angegeben. Mit dem Test-OAuth-Token kann testweise auf die https://interseroh-portal-test.com/entsorger-bestellung/v1/stammdaten zugreifen.

Eine Online-Dokumentation zum genauene Aufbau des Request sowie der Response ist unter der URL http://[interseroh-host]/entsorger-stammdaten/v2/api-docs?group=stammdaten verfügbar.

4.2 Stammdaten (Lieferant)

Dieser Typ des Stammdatenservices gibt alle dem Benutzer zugeordneten Lieferanten, sowie deren Betriebsstätten zurück. Alle anderen Objekte sind mit null belegt.

Request

Das Abrufen von Stammdaten wird als GET-Request ausgeführt. Der Aufruf der URL gestaltet sich so: https://[interseroh-host]/entsorger-stammdaten/v1/stammdaten?typ=Lieferant

Der notwendige Parameter typsteht in diesem Fall Lieferant, wobei Groß-/Kleinschreibung keine Rolle spielt.

Response

Bei einem Fehler wird der entsprechende HTTP-Statuscode zurückgesendet. Ist der Statuscode 200, befinden sich die Nutzdaten im Body der Response im JSON-Format.

Im Fall eines Erfolges wird die Antwort im JSON-Format zurückgegeben, der den entsprechenden Stammdatensatz enthält. Der Stammdatensatz für den Typ LIEFERANT hat den folgenden Aufbau:
